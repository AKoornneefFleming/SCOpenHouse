Additional Details:

One of the solutions that Safari Consulting has delivered is a file geodatabase library (SAWC FGDB Library) to house all layers of the project and facilitate good database management practices. We originally designed this approach to be flexible to help adapt to changes in the database design and structure throughout the project. The other benefit of separating data into specific and intuitive locations was to reduce the likelihood of data being duplicated in the future.

Methodology - File Geodatabase

Before collecting any data on-site, the team sat down with the client and reviewed the data which already existed, finalized the list of data to be collected, and determined exactly what attributes were to be included per layer. We maximized our time at SAWC by building the geodatabase with empty layers before arriving on-site (see Deliverable 1).

All the data collected on-site was then cleaned up and transferred to its appropriate file geodatabase in the SAWC FGDB library, created by Safari Consulting. After reviewing all the data and discussing further the needs of the client, we decided the best approach was to create two separate file geodatabases. The first was to hold all the data that was intended for public viewing. The second was to hold all the sensitive data, which should only be accessed by members of SAWC.

After cleaning and organizing all the data into the proper file geodatabases, we further organized the data into feature datasets, where it was applicable. For example, in the staff file geodatabase all the critical infrastructure and life safety data were placed in their own feature datasets. We also applied best practices for naming conventions for all the layers within the file geodatabases as well as field names within each layer. We did this by excluding spaces, periods, or special characters in the names, and not starting any name with a number. Additionally, we kept the names singular instead of plural i.e., “Road” instead of “Roads” and changed all the layer and field aliases which allowed for a cleaner display in the legends and pop-ups. For example, we would name layers to be intuitive to the data, such as “CI_SewageSepticTank”, whereas the alias would simply display the layer as “Septic Tank”. Furthermore, we set field lengths to ensure the most efficient storage possible.

The first steps in standardizing the FGDB.

Finally, we created relationship classes between layers (such as between buildings and rooms), created domains and made sure to assign the domains to the proper fields.

Setting domains.

Uploading Layers from the FGDB to ArcGIS Online

These feature class layers were uploaded to AGOL and hosted as feature layers. These layers were then used to create web maps within AGOL and those maps were used in applications such as ArcGIS Field Maps and Experience Builder.

Uploading a layer to AGOL.

Once these feature layers were shared to AGOL, we were able to overwrite them to push through any updates. Uploading features to AGOL from ArcGIS Pro allowed us to customize the symbology within ArcGIS Pro, which was maintained in the layers once uploaded. This was very useful as when changes were made to the data and we needed to re-upload or overwrite layers, we did not have to re-do any symbology as it was carried forward to AGOL.

Overwriting layers to AGOL.

Results

The final result of our file geodatabase deliverable were two separate file geodatabases. We have also included a file geodatabase that was given to us by the client that holds general data about Africa, such as administrative boundaries. All three of these file geodatabases have been placed in their own library folder, which we have called “SAWC_FGDB_Library”.
